A. How gpf depends on gain

Declaration shape

gain-core is a plain dependency name in two places:

  • core/pyproject.toml:31
  • web_api/pyproject.toml:28

The committed pyproject.toml:37–48 deliberately leaves gain-core out of the [tool.uv.sources] block. uv resolves it like any package, but the only place a wheel exists is dist/gain/, so consumers point uv there via --find-links ./dist/gain.

Lockfile pin

uv.lock records the exact resolved version (e.g. 2026.5.2.post1.dev27+gc0d9fe7f1) — a hatch-vcs-derived string that encodes the gain SHA. The pin is a soft anchor: it represents “the gain I last tested gpf against,” not “the gain I will build against.”

A developer updates it via: 

cd ../gain && uv build --package gain-core --out-dir ../gpf/dist/gain
cd ../gpf && uv lock --upgrade-package gain-core --find-links ./dist/gain

and commits the new lockfile. Meta-repo bump commits with the title Bump gpf: ... — uv.lock: gain-core +<SHA> record this pairing.

Why --upgrade-package is everywhere in CI

Plain uv sync --find-links ./dist/gain honors the lock. If dist/gain/ contains a different wheel than the lockfile pins, sync fails. --upgrade-package gain-core tells uv: “re-resolve only this package against --find-links, ignore the lock entry.” Every CI Dockerfile carries this flag:

  • core/Dockerfile:49,54
  • web_api/Dockerfile:53,58
  • federation/Dockerfile:48,53
  • rest_client/Dockerfile:47,52

B. How Jenkins builds for gpf refer to gain

gpf master CI — floating gain

Jenkinsfile:295–331 defines the Fetch gain wheel stage: 

copyArtifacts(
    projectName: 'iossifovlab/gain/master',
    selector:    lastSuccessful(),
    filter:      'dist/core/*.whl',
    target:      'dist/gain',
    flatten:     true,
)

Every gpf master build pulls the most recent green gain master build. There is no read of .gitmodules, no read of gpf's uv.lock, no SHA matching of any kind.

Consequence: a gain master push can break gpf master CI with zero gpf changes. The lockfile pin offers no protection because CI's --upgrade-package gain-core bypasses it.

gpf release — deterministic per gpf tag

Jenkinsfile.release:402–497 has two paths in Fetch upstream artefacts:

Default path (tag-push dispatcher)

copyArtifacts(
    projectName: 'iossifovlab/gpf/master',
    selector:    specific(env.UPSTREAM_BUILD_NUMBER),
    filter:      'dist/base-images.lock, dist/gain/*.whl',
)

The release pipeline finds the gpf-master CI build whose GIT_COMMIT matches the tagged commit, and copies that build's dist/gain/*.whl. Pulls the same gain wheel upstream gpf-master tested with for that commit. Deterministic per gpf release — but the bundled gain version is a dev+SHA string, not a clean CalVer.

Pinned path (GAIN_PIN_VERSION=2026.5.12, manual UI only)

Curls the CalVer-tagged wheel from wheels.seqpipe.org/gain/ instead. Sharp tool: the gpf release is not re-tested after the swap; the operator certifies API compatibility.

Production image — wheels-only bundle

web_api/Dockerfile.production:48–78: 

COPY dist/gain/gain_core-*.whl ./
RUN uv pip install --only-binary=:all: --no-binary=mysqlclient /wheels/*.whl

The released registry.seqpipe.org/gpf-web:<TAG> image bundles the gain wheel inside its /opt/gpf venv. No network call to gain at install time — the wheel is baked in.

C. How to develop gpf locally

Two modes — pick one per checkout.

CI shape (wheel) Editable
What's installed Wheel from dist/gain/, rebuilt on every gain SHA change pip install -e ../gain/core — gain source is live
After git pull in gain cd ../gain && uv build --package gain-core --out-dir ../gpf/dist/gain && cd ../gpf && uv sync --upgrade-package gain-core --find-links ./dist/gain uv sync — picks up gain's current working tree, no rebuild
Config files Untouched (CI shape) pyproject.toml has gain-core = { path = "../gain/core", editable = true } inline in the existing [tool.uv.sources] block; uv.lock has source = { editable = "../gain/core" }
Safety Always safe to commit Both files must be skip-worktree — committing them breaks CI

Activating the editable override

Add the path source inline inside the existing [tool.uv.sources] block in pyproject.toml: 

[tool.uv.sources]
gpf-core = { workspace = true }
gpf-web = { workspace = true }
gpf-federation = { workspace = true }
gpf-rest-client = { workspace = true }
gain-core = { path = "../gain/core", editable = true }

Then: 

git update-index --skip-worktree pyproject.toml uv.lock
uv sync
Don't use the subheader form. Writing the override as a separate [tool.uv.sources.gain-core] table (the form the README's tomlkit merge script produces) can be silently normalized away by a later uv sync. The inline-assignment form above survives sync intact.

Cross-repo PR ordering

When a change needs both gain and gpf code:

  1. Open and merge the gain PR. Until gain master goes green, gpf CI may transiently break — even unrelated gpf branches.
  2. Wait for gain master CI to publish a green build. This is what gpf's Fetch gain wheel stage reads.
  3. Open the gpf PR.

While iterating locally, the editable override lets you test both halves of the change together before either PR exists.

Pre-PR ritual

Before pushing a gpf branch — especially one that exercises new gain surface — reproduce the CI shape locally: 

# 1. Revert to CI shape
git update-index --no-skip-worktree pyproject.toml uv.lock
git checkout -- pyproject.toml uv.lock

# 2. Rebuild gain wheel from gain's HEAD, re-resolve gpf
cd ../gain && uv build --package gain-core --out-dir ../gpf/dist/gain
cd ../gpf && uv sync --upgrade-package gain-core --find-links ./dist/gain

# 3. Run tests under the CI shape
uv run pytest -v tests/small/

# 4. If uv.lock changed, commit it (the "Bump gpf ... +gSHA" workflow).
#    If not, your branch doesn't need to bump the gain pin.

# 5. Restore editable for ongoing dev
#    (re-add the inline source, re-skip-worktree both files, uv sync)

The genomics-toolbox meta-repo role

Purely a human coordination record. Nothing in CI reads .gitmodules or the meta-repo's submodule pins. The commit pattern Bump gpf: ...@<gpf-SHA> — uv.lock: gain-core +<gain-SHA> paired with an earlier Bump gain: ...@<gain-SHA> documents “these two SHAs are a tested combination.”

Useful for:

  • Reproducible clones: git clone seqpipe/genomics-toolbox && git checkout <meta-SHA> && git submodule update --init --recursive
  • Documenting what got deployed
  • Shaping ../gain/-relative editable workflows

But not for triggering or pinning CI builds.
